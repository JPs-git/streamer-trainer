# 基于主播语音的实时弹幕生成系统 — 架构设计

**版本**：1.0  
**日期**：2026-06-04  
**依据**：[用例文档 v1.1](./use-cases-v1.1.md)

---

## 1. 系统概述

为 OBS 直播场景设计的主播培训辅助系统。通过捕获主播语音，生成由多个具有独立身份和记忆的虚拟观众驱动的实时弹幕，营造真实互动氛围，训练主播的直播能力。

---

## 3. 核心组件设计

### 3.1 WebSocket 连接

三条独立 WS 连接，职责分离：

| 连接 | 方向 | 协议 | 说明 |
|------|------|------|------|
| `audio_in` | OBS音频插件 → 后端 | 二进制PCM帧 + JSON控制帧 | 主播语音流 |
| `danmaku_out` | 后端 → OBS Browser Source | JSON文本帧 | 弹幕/入场/离场通知 |
| `control` | 双向 | JSON文本帧 | 配置控制 |

**danmaku_out 帧格式：**

```json
// 弹幕消息
{"type":"danmaku","id":"xiaobing","name":"小冰",
 "text":"这游戏怎么玩呀？","personality":"curious",
 "effect":"normal"}

// 入场通知
{"type":"system","action":"enter","name":"小冰"}

// 离场通知
{"type":"system","action":"leave","name":"老王"}

// 特殊弹幕（压力型高亮）
{"type":"danmaku","effect":"highlight","id":"tuzi",
 "name":"兔子","text":"你说得不对！","personality":"aggressive"}
```

---

### 3.2 ASR 模块

- **模型**：Whisper（本地部署，如 `whisper.cpp` 或 `faster-whisper`）
- **输入**：WebSocket 收到的 PCM 音频帧
- **输出**：文本（带时间戳偏移量）
- **策略**：流式处理，每检测到一句话结束就输出结果，而非等整句说完

---

### 3.3 虚拟观众管理器

#### 角色数据结构

```python
@dataclass
class ViewerMemory:
    streamer_log: list[dict]       # 主播全部发言 [{text, offset}]
    my_danmaku: list[dict]         # 自己发过的弹幕 [{text, offset, directed_to}]
    other_danmaku: list[dict]      # 他人弹幕摘要 [{from, to, summary}]
    relationships: dict[str, str]  # 角色间关系动态演化

@dataclass
class VirtualViewer:
    viewer_id: str
    name: str
    persona: str               # 人设描述
    personality_type: str      # curious/cheerful/aggressive/bystander
    state: str                 # 状态机状态
    memory: ViewerMemory
    entry_time: int
    last_active: int
    interaction_count: int
```

#### 状态机生命周期

```
未激活池 ──调度──→ NEW (刚入场)
                     │
                     ▼
                 WATCHING (常规观看)
                  /         \
                 ▼           ▼
            LEAVING (准备离场)  WATCHING (可维持)
                 │
                 ▼
            COOLDOWN (冷却池，可再次调度)
```

#### 入场/离场调度规则

- 初始活跃人数：3-5 人
- 稳态：4-8 人同时在线
- 轮换周期：每 3-5 分钟新人入场，老人离场
- 冷却期：离场角色 5-10 分钟后可重新登场（模拟"回来了"）

#### 角色类型

| 类型 | 角色标签 | 颜色 | 行为特征 |
|------|---------|------|---------|
| 引导型 | curious | 蓝色 `#4FC3F7` | 提问、求演示、制造话题 |
| 捧场型 | cheerful | 绿色 `#81C784` | 夸赞、喊666、正向反馈 |
| 压力型 | aggressive | 橙色 `#FF8A65` | 反驳、质疑、压力测试 |
| 旁观型 | bystander | 灰色 `#B0BEC5` | 附和、颜文字、路过 |

其中压力型角色是核心训练价值所在，设计原则是**建设性的对抗**——有逻辑地质疑而非无脑喷。

#### 角色间社交

- 角色A提问 → 角色B（非主播）可回答 → 角色A致谢（形成角色间闭环）
- 角色C反驳主播 → 角色D（捧场型）可维护主播
- 角色间关系动态演化，影响后续互动方向

---

### 3.4 选择器（Stage 1）

**输入**：
- 当前 ASR 文本
- 所有活跃角色状态摘要（每人一行，含当前状态和关注方向）

**输出**：JSON 格式的发言角色列表 + 意图

```json
[
  {"id": "xiaobing", "intent": "作为新人询问新手友好度"},
  {"id": "laowang", "intent": "反驳新手五分钟上手的说法"}
]
```

**特性**：使用轻量模型（可复用云端或独立小模型），token 消耗低。

---

### 3.5 生成器（Stage 2）

**机制**：每个选中角色获得**完全隔离的专属 prompt**，并行调用云端 LLM。

**prompt 结构**：

```
[角色档案]
昵称: 小冰
人设: 好奇宝宝
类型: curious
语言风格: 多用"呀/吗/呢"，颜文字

[完整记忆]
主播说过的话: ...
自己发过的弹幕: ...
其他人的弹幕摘要: ...
角色间关系: ...

[当前主播说话]
"..."

[任务]
以角色身份发一条符合当前状态的弹幕。
```

**并行性**：所有选中角色的生成请求同时发出，总延迟 ≈ 单个最慢生成（2-3 秒）。

### 3.6 H5 前端

**技术选型**：纯 HTML + CSS + JS（OBS 内置 Chromium 运行）

**展示形式**：竖屏向上滚动的聊天列表窗口

```
┌────────────────────────┐
│ 直播间聊天         ⚙️   │
├────────────────────────┤
│ 🟢 小冰 进入了直播间    │
│ 🗣 小冰: 今天播什么？   │
│ 🗣 阿强: 刚来就有好戏   │
│     看？               │
│ ⚠ 兔子: 这操作我上也   │
│     行                 │
│ 🗯 静静: 哈哈哈+1      │
│ 🗣 小冰: 这游戏好玩吗？ │
│ 🗣 阿强: @小冰 好玩    │
│ 🔴 老王 离开了直播间    │
└────────────────────────┘
         ↕ 自动滚动
```

**交互设计**：
- 弹幕从底部进入，向上滚动
- 自动滚动到最新消息
- 鼠标悬停暂停滚动
- 角色昵称按类型着色
- 入场/离场/@提及有不同前缀标识
- 可选 OBS 插件设置面板：密度、窗口大小、位置

## 4. 数据流（一轮完整闭环）

```
1. OBS 音频插件捕获 PCM → WS audio_in
2. 后端 ASR (Whisper) → 文本（带时间戳）
3. 文本追加到所有活跃角色的 memory.streamer_log
4. 选择器：ASR文本 + 各角色状态 → 选中角色ID + 意图
5. 生成器：为每个选中角色构建独立prompt，并行调用LLM
6. LLM返回 → 弹幕文本
7. 更新各角色 memory.my_danmaku / other_danmaku / last_active
8. 角色间互动时更新 memory.relationships
9. 弹幕通过 danmaku_out WS 推送到 H5 前端
10. H5 前端渲染展示
```

## 5. 非功能性设计

### 延迟要求
- 从语音到弹幕展示：目标 3 秒内
- 瓶颈在 LLM 生成（2-3秒），ASR 和 WS 传输合计 < 1 秒

### 角色记忆上限
- 单场直播内，per-role memory 随直播时长线性增长
- 策略：当 streamer_log 超过阈值时，对早期内容做摘要压缩，保留最新 N 条原文

### 并发
- 单主播场景，单进程即可
- 选择器和生成器调用异步并行处理
